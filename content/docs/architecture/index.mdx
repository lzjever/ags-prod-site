# 整体架构总览

AgentSmith 的架构并不是把若干页面拼成一个工作台，而是把企业使用 AI 时最容易分裂的几件事重新组织到同一套结构中: 业务团队如何进入系统、项目如何承接日常工作、模型与智能体如何被统一调用、结果如何沉淀为资产、限制与审计又如何在同一条链路上持续生效。

<ValueStatement

  summary="对 AgentSmith 来说，真正重要的不是用户能看到多少模块，而是这些模块是否围绕同一个项目边界持续工作。"

<SiteSection
  eyebrow="Three Layers"
  title="这套产品先分层，再在项目内收口"
  description="System、Workspace、Project 三层结构解决的是边界问题，而不是菜单问题。"
>

`System` 只负责系统级控制平面，例如工作区生命周期、身份提供商配置和开通状态。`Workspace` 负责租户或业务域边界，让用户先进入正确的业务空间，再进入后续项目。`Project` 则承接真正的工作对象: 对话、任务、文件、智能体、模型资源、成员和审计证据都围绕它持续组织。

这种结构的好处在于，系统管理和业务使用不会互相打断。管理员可以在系统侧开通工作区，业务团队随后进入工作区、创建项目，并在同一个项目中完成使用和治理，而不需要在多个彼此脱节的后台之间来回切换。

</SiteSection>

<SiteSection
  eyebrow="Use And Govern"
  title="Use 面与 Govern 面不是两套系统"
  description="它们面向不同任务，但共享同一个项目对象、同一组资源和同一条证据链。"
>

Use 面负责让团队把 AI 真正用起来。Chat 提供最快的交互入口，Notebook 提供面向任务的执行界面，Files 提供长期可复用的项目资产层，Agents 则把外部或内部执行能力接入到项目中。Govern 面负责让这套使用方式能够长期运行。Endpoints 定义模型资源，Resource Policy 解释访问限制，Credentials 承接连接信息，Members 负责协作边界，Usage 呈现消耗状态，Audit 保留回看与追溯依据。

这两组能力并不是彼此隔离的。Chat 和 Notebook 会使用项目中定义好的 Endpoint；Notebook 的执行又会产出 trace 和 artifacts；这些记录会继续进入 Usage 和 Audit。也正因为使用面和治理面共享同一组对象，AgentSmith 才更像一套系统，而不是“前台 + 后台”的松散组合。

</SiteSection>

<SiteSection
  eyebrow="Execution Path"
  title="界面、后端和执行环境通过同一条运行链路连接"
  description="从代码和测试来看，AgentSmith 已经具备比较清晰的执行骨架。"
>

当前实现中，前端应用负责页面与交互状态，项目壳层通过统一 sidebar 将 `overview`、`chat`、`notebook`、`files`、`agents`、`endpoints`、`resource-policy`、`credentials`、`members`、`usage`、`audit`、`settings` 等模块组织到同一套项目工作台中。后端的 API entry 负责鉴权、路由、权限判断和协议编排，再把请求分发到项目、文件库、治理、聊天和任务执行等具体处理链路。

在执行层，Notebook 任务并不是一次前端按钮点击后的黑盒请求。执行前会检查 agent 可用性、endpoint 状态和治理预检；执行时会携带 `workspace_id`、`project_id`、`task_id`、`run_id`、`endpoint_id`、输入引用和凭据文件等上下文；运行过程中会持续回传 `message`、`trace_event`、`artifact`、`done/error`；执行结束后，使用量与审计事件会被统一记录下来。

</SiteSection>

<SiteSection
  eyebrow="Why This Matters"
  title="这套架构真正解决的不是功能堆叠，而是企业 AI 的结构性问题"
  description="很多团队并不缺入口，真正缺的是统一边界、统一资源和统一证据。"
>

如果 Chat、任务执行、文件管理和治理证据各自生长，团队最终会面对几套彼此不一致的系统: 一套看起来好用，一套负责限制，还有一套只在出问题时才被打开。AgentSmith 选择的是另一条路: 让项目成为唯一的业务容器，让资源、执行和证据始终围绕这个容器工作。这样一来，团队可以先把 AI 用起来，再把它稳定地运行下去，而不是在不同阶段被迫更换工作方式。

</SiteSection>
