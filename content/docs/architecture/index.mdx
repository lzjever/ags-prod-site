# 整体架构总览

从产品视角看，AgentSmith 最值得理解的并不是页面数量，而是它如何把使用面、治理面和执行链路组织成一套清晰系统。

  eyebrow="Architecture"

  presets={['project-overview-guide', 'project-endpoints-guide']}

这套结构可以概括为五个层次: `System Admin`、`Workspace`、`Project`、`Use 面` 和 `Govern 面`。它们并不是上下脱节的管理层级，而是共同决定了团队如何在同一业务边界内使用 AI、运行智能体并持续完成治理。

<SiteSection
  eyebrow="Product Skeleton"
  title="项目容器是整个系统的共同骨架"
  description="Use 与 Govern 不是两套系统，而是在同一个项目容器中并列切换。"
>

`System Admin` 负责工作区生命周期和底层基线，`Workspace` 则提供真实租户边界，让业务空间先被清楚划定。真正承接日常工作的，是 `Project` 这一层: 它既容纳 Chat、Notebook、Files、Agents 这些使用面，也容纳 Endpoints、Policy、Credentials、Members、Usage、Audit、Settings 这些治理能力。也正因为它们共同收拢到项目容器之中，AgentSmith 才不会演化成两套互不相干的系统。

</SiteSection>

<SiteSection
  eyebrow="Execution"
  title="前端界面、执行协议与治理能力通过统一链路相连"
  description="AgentSmith 的差异化之一，是把任务执行、资源治理与审计回看组织在同一系统里。"
>

前端界面承载的是用户交互与项目上下文，API 入口层则负责鉴权、路由和执行编排，而智能体会通过协议化事件把 delta、trace、artifact 和 done/error 持续回传。Notebook 的任务记录、Usage 的消耗事实和 Audit 的回看依据因此能够彼此对齐，不再分散在不同系统里。这正是 AgentSmith 架构层面最有价值的地方之一。

</SiteSection>
