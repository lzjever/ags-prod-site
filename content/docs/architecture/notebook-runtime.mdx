# Notebook 运行链路与智能体执行环境

Notebook 是 AgentSmith 中承接长期任务的核心界面。它会把任务、输入、conversation、trace 和 artifacts 组织在同一个工作流中，让团队能够持续查看执行过程、复用输入资料并保留输出结果。

<SiteSection
  eyebrow="Notebook Task Model"
  title="Notebook 用任务对象组织较长链路的智能体执行"

>

在 Notebook 中，一次执行不会只留下最后一句回复。系统会保留任务本身、输入来源、消息过程、trace 事件和输出产物，使后续成员能够继续理解、继续推进或继续复用。这一点对复杂分析、内容生产和工程辅助类任务尤为重要，因为这些工作往往需要更长上下文，也需要更明确的结果沉淀方式。

对通用智能体而言，Notebook 还承担了统一操作界面的职责。用户不需要先学习命令行运行方式，再手动准备目录、复制文件和检查输出位置，而是可以在同一个任务界面里完成准备输入、发起执行、查看过程和保存结果。

</SiteSection>

<SiteSection
  eyebrow="What A Task Contains"
  title="一个 Task 里通常会同时保留五类信息"
  description="理解这五类对象之后，Notebook 为什么适合长期任务就会更清楚。"
>

第一类是任务本身，例如标题、所属项目、当前状态和最后活动时间。第二类是输入资料，当前可以来自项目文件、历史产物或 URL。第三类是消息过程，也就是任务中的提问、回答和多轮推进。第四类是 trace，用来反映执行过程中的关键步骤和状态。第五类是 artifacts，也就是任务真正生成出来的文件、图片、报告或其他产物。

这些对象会被持续组织在一起，所以 Notebook 不是“只看最后一条回复”的地方，而是“回看整项工作”的地方。

</SiteSection>

<SiteSection
  eyebrow="Runtime Assets"
  title="一次 Notebook 执行会生成一组清晰的运行时资产"
  description="这些对象共同构成了任务的执行上下文。"
>

当前实现已经具备比较明确的任务运行时模型。系统会为任务解析输入引用，准备 `task workspace` 工作目录，生成 `.mbos/task-inputs.json` 输入清单，并在工作目录中创建 `artifacts/` 目录用于收集执行产物。同时，任务工作区还会生成 `AGENTS.md` 等辅助文件，明确 headless 执行约束和输入获取方式。

这里涉及几个关键对象。`task workspace` 是一次任务真正运行的工作目录；`.mbos/task-inputs.json` 是这次任务的输入清单；`artifacts/` 是执行产物的固定输出目录；`AGENTS.md` 是系统自动生成的运行规则说明，告诉 runner 这是一个无界面的任务环境，输入资料在哪里，输出文件应该放到哪里。

Notebook 不只是一个页面表单。任务输入从哪里来、产物落到哪里、如何取回附加文件、任务如何被后续再次引用，都会在系统中以明确对象的方式被组织起来。

</SiteSection>

<SiteSection
  eyebrow="Preflight"
  title="执行开始前，系统会先完成资源和治理预检"
  description="这一步确保任务不是“点了就跑”，而是在可用资源和可解释限制内运行。"
>

Notebook 任务在正式执行前，会检查目标智能体是否可用、绑定的 Endpoint 是否可访问，并执行治理预检。资源策略、速率限制和支出限制都会在这里被检查；如果不满足条件，系统会明确终止执行，并把失败原因写入后续证据链。

这类预检沿用了项目级治理规则，因此 Notebook 的任务执行与项目中的资源和策略体系保持一致。

</SiteSection>

<SiteSection
  eyebrow="Execution Paths"
  title="External Agent 与 Internal Sandbox 是两条不同但兼容的执行路径"
  description="它们共享任务上下文，只是在运行环境上有所区别。"
>

在 external agent 路径下，执行者已经运行在平台之外。它可以是本地 runner，也可以是企业自己维护的外部进程。AgentSmith 的职责是把任务通过 WebSocket 发过去，并持续接收 delta、trace 和 artifact 事件。这条路径更适合接入已有执行程序，也是当前更成熟的一条主链路。

在 internal agent 路径下，平台会先准备托管执行环境，再把 runner 放进去执行。现有代码里，这一步会先触发 sandbox starting，再通过 internal agent pod manager 去 ensure 运行环境已经就绪。按 PRD 的目标形态，这条路径会把执行放入 K8s / Pod 一类更可控的托管环境中，让平台逐步承担启动、保活和回收责任，而不是让用户自己维护宿主进程。

这条托管路径当前仍在继续收敛，但它已经明确表达了产品方向: 通用智能体最终不必长期依赖用户本地宿主环境，而可以进入更可控的执行环境。

无论使用哪条路径，Notebook 都会保持同样的任务视图。用户不需要切换到另一套完全不同的操作方式，才能查看执行状态或结果。

</SiteSection>

<SiteSection
  eyebrow="Managed Execution"
  title="Internal Agent 托管到 K8s 后，和 External Agent 的差别会更明显"
  description="两条路径都能跑任务，但平台承担的责任不同。"
>

如果把 internal agent 放到 K8s / Pod 这类托管环境中，平台就不只是一个“转发请求”的系统了。它还要负责准备容器或 Pod、确认运行环境可用、在任务存活期间保活，以及在结束后回收或释放资源。对于企业来说，这通常意味着更好的环境隔离、更稳定的运维边界和更一致的执行方式。

相比之下，external agent 更像一条开放接入路径。它的优势是灵活，适合快速连接已有 runner；它的代价是，运行环境本身仍主要由平台外部维护。两者并不是互斥关系，而是分别服务于不同阶段和不同部署条件。

</SiteSection>

<SiteSection
  eyebrow="Project Continuity"
  title="输入资料、执行过程和结果会在项目中持续保留"
  description="团队可以围绕同一项任务继续查看、接手和复用后续成果。"

输入文件、凭据、运行目录和结果文件不会停留在个人环境中，而是逐步进入项目结构，并与 Files、Artifacts、Usage 和 Audit 保持连续关系。这样一来，执行结果既可以被当前任务使用，也可以成为后续工作的输入和参考。

</SiteSection>
