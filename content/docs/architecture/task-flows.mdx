# 关键任务流

这一页汇总了 AgentSmith 中最重要的几条工作路径。你可以从这里看到团队如何进入系统、如何在项目中发起工作，以及执行结果如何继续进入资源管理、用量观察和审计回看。

<ValueStatement
  eyebrow="Task Continuity"
  title="工作区、项目、任务和审计记录会沿同一条路径持续衔接。"
  summary="团队从进入工作区开始，到创建项目、运行任务和查看证据，都围绕同一组对象继续推进。"
  detail="这让日常使用、长期任务和治理回看能够保持一致。"
/>

<SiteSection
  eyebrow="Mainline"
  title="系统主线从 Workspace 开通开始，在 Project 中形成持续工作"
  description="这是所有使用面和治理面最终共享的一条总路径。"
>

系统管理员先创建并发布 Workspace，让业务团队获得可用入口。随后，工作区管理员或具备创建权限的成员建立 Project。成员进入项目后，开始使用 Chat、Notebook、Files 和 Agents 完成日常工作，执行过程中的结果、trace、usage 和 audit 事件也会持续沉淀。管理员再基于这些事实完成资源限制、排查和追溯。

团队不需要先在一个地方完成工作，再到另一个地方查看限制或回看结果。工作区、项目、任务和审计记录会在同一套系统中持续衔接。

</SiteSection>

<SiteSection
  eyebrow="Daily Work"
  title="普通成员最常见的路径，是从 Chat 或 Notebook 开始"
  description="Use 面并不是两条孤立入口，而是轻量交互与长任务执行的分工。"
>

当用户需要快速提问、分析或连续对话时，通常会从 Chat 开始。这里强调的是低门槛和快速反馈。当工作需要更长链路、更明确输入、更完整结果沉淀时，用户会进入 Notebook，把任务、输入、执行过程和产物组织为一条长期记录。Files 则贯穿这两条路径，为对话和任务提供上下文输入，并承接后续结果沉淀。

这种分工让产品既能满足“快速开始”的需要，也能承接“复杂任务持续推进”的需要，而不会把所有工作都塞进一个过度复杂的界面。

</SiteSection>

<SiteSection
  eyebrow="General Agent"
  title="通用智能体运行流是当前最关键的一条产品链路"
  description="它体现了 AgentSmith 不只是一个聊天入口，而是一套执行组织系统。"
>

用户进入项目后，会先打开 Notebook，选择或创建一个 Task，再把项目文件、URL 或历史 Artifact 作为输入加入上下文。接下来，系统根据任务绑定的 Agent 和 Endpoint 完成资源与权限预检，准备任务工作目录、输入清单、凭据文件和技能挂载，然后由 external agent 或 internal agent 执行。执行过程中，conversation、trace 和 artifacts 持续回传，最终结果既保留在任务历史中，也可以沉淀到项目文件库。

这条链路会把运行前、运行中和运行后的关键对象都放进项目结构中管理。用户不需要单独管理本地目录、结果文件和执行日志，团队也可以围绕同一个任务继续协作。

</SiteSection>

<SiteSection
  eyebrow="Two Execution Paths"
  title="同一条任务流，可以落到两种执行路径"
  description="差别主要在运行环境由谁负责。"
>

如果任务绑定的是 external agent，系统会把请求发给已经在平台外部运行的执行端，并持续接收结果回传。这个执行端可以是本地 runner，也可以是企业自己维护的服务进程。用户通常能更快开始使用，因为不需要平台先准备托管环境。

如果任务绑定的是 internal agent，系统会先准备托管运行环境，再把 runner 放进去执行。这条路径的目标，是让平台逐步承担执行环境的准备、保活和回收工作。按当前 PRD 和代码骨架，它会依赖 sandbox manager 与 Pod 级托管能力，在基础设施到位时走向更受控的执行方式。

</SiteSection>

<SiteSection
  eyebrow="Blocked Runs"
  title="有些任务不会进入执行阶段，而是会在预检阶段被明确挡下"
  description="这不是系统失败，而是治理规则在起作用。"
>

如果用户没有访问某个 Endpoint 的权限，或者当前请求已经命中速率限制、支出限制，系统会在执行真正开始前直接终止任务。对用户来说，这类情况会以明确原因体现出来，而不是伪装成普通的模型报错。对团队来说，这意味着资源限制是实际生效的，而且后续可以在 Usage 和 Audit 中回看这些记录。

</SiteSection>

<SiteSection
  eyebrow="Governance"
  title="治理任务流不是另一条平行线，而是使用结果的继续处理"
  description="Govern 面真正关注的是资源、限制和证据是否与实际使用对齐。"
>

项目管理员会先准备 Endpoint、Credentials、Members 和资源策略，让资源范围和协作边界先被定义清楚。随着 Chat、Notebook 和 Agents 被实际使用，Usage 会持续记录消耗状态，Audit 会持续记录关键变更和异常事件。管理员随后再基于这些事实判断限制是否需要调整、异常是否需要排查、成员和资源边界是否需要修正。

也就是说，治理并不是先独立存在、再等待业务使用，而是使用结果不断回流到治理视图中。正因为如此，Usage 和 Audit 才不是普通统计页，而是项目运行情况的事实层。

</SiteSection>

<SiteSection
  eyebrow="End-To-End Use"
  title="从进入工作区到查看执行结果，团队会沿着同一条工作路径继续前进"
  description="工作空间、项目、任务、文件、资源和审计记录都会在这条路径中持续对应。"
>

成员进入工作区后，可以在项目中发起对话、组织 Notebook 任务、使用文件库并调用智能体。执行结果会继续沉淀到项目历史、文件资产、用量记录和审计事件中，便于后续协作、限制调整和异常排查。

</SiteSection>
