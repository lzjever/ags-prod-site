# Agents

Agents 模块负责把执行能力正式接入项目。它的意义不只是列出有哪些智能体，而是让外部智能体和平台托管智能体都能以统一方式进入团队工作，让成员知道有哪些执行能力可用、它们适合做什么，以及执行结果最后会回到哪里。

<SiteSection title="它解决什么问题" description="很多团队已经有 runner 或脚本，但这些能力常常难以真正进入团队协作。">

如果智能体只存在于某个人的本地环境里，团队成员往往很难知道它是否在线、能做什么、适合在什么任务中使用，也很难在后续回看过程中把执行结果与具体项目对应起来。Agents 模块解决的，就是把这些原本偏个人化、偏脚本化的能力升级成项目中的正式对象。

</SiteSection>

<SiteSection title="External Agent 与 Internal Agent 有什么不同" description="两者都能执行任务，但谁来提供运行机器、谁来维护执行进程并不一样。">

External Agent 指已经运行在平台外部的执行者，例如开发者电脑上的 runner，或者企业自己维护的服务进程。AgentSmith 会把任务发给它，再接收消息、trace 和产物回传。换句话说，任务是在你已经准备好的机器上执行的，平台负责把任务送过去并接回结果。

Internal Agent 则代表另一条方向：平台在需要执行时自己准备受控的执行环境，再把 runner 放进去执行。在当前产品方向里，这条托管路径会逐步走向 K8s / Pod 这类更可控的运行方式。换句话说，任务不是跑在某个人手工维护的机器上，而是跑在平台统一准备和回收的环境里。

对用户来说，最重要的区别在于运行机器由谁提供、执行进程由谁维护。External Agent 更适合快速接入已经存在的能力；Internal Agent 更适合希望把执行环境收回平台统一管理的团队。

</SiteSection>

<SiteSection title="它如何进入整体工作流" description="Agents 不是孤立模块，而是任务链路中的关键节点。">

在典型工作流中，管理员先在项目中接入和管理智能体，成员再在 Chat 或 Notebook 中选择合适的执行能力。任务启动后，系统会继续组织上下文、检查资源限制、接收过程回传，并把结果留在任务历史、文件资产和审计记录中。也就是说，Agents 页面不是“登记一下智能体列表”，而是把执行能力真正接到项目工作过程中的关键入口。

</SiteSection>
