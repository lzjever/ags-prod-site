# Agents

Agents 模块负责把执行能力正式接入项目。它的意义不只是列出有哪些智能体，而是让外部智能体和平台托管智能体都能以统一方式进入团队工作流，进而被 Chat、Notebook 和后续治理链路持续使用。

<SiteSection title="它解决什么问题" description="很多团队已经有 runner 或脚本，但这些能力常常难以真正进入团队协作。">

如果智能体只存在于某个人的本地环境里，团队成员往往很难知道它是否在线、能做什么、适合在什么任务中使用，也很难在后续回看过程中把执行与项目上下文对应起来。Agents 模块解决的，就是把这些原本偏个人化、偏脚本化的能力升级成项目中的正式对象。

</SiteSection>

<SiteSection title="External Agent 与 Internal Agent 有什么不同" description="两者都能执行任务，但接入方式和托管方式不同。">

External Agent 指已经运行在平台外部的执行者，例如本地 runner 或企业自己维护的服务进程。AgentSmith 通过统一协议把任务发给它，并接收消息、trace 和产物回传。Internal Agent 则代表另一条方向: 平台在需要执行时自己准备受控运行环境，再把 runner 放进去执行。在当前产品方向里，这条托管路径会逐步走向 K8s / Pod 这类更可控的运行环境。

对用户来说，最重要的区别在于运行环境由谁承担。External Agent 更适合接入已有能力，Internal Agent 更适合逐步走向平台托管和更稳定的企业运行方式。

</SiteSection>

<SiteSection title="它如何进入整体工作流" description="Agents 不是孤立模块，而是任务链路中的关键节点。">

在典型工作流中，管理员先在项目中接入和管理智能体，成员再在 Chat 或 Notebook 中选择合适的执行能力。任务启动后，系统会继续组织上下文、检查资源限制、接收过程回传，并把结果留在任务历史、文件资产和审计记录中。也就是说，Agents 是把执行能力接到项目工作流中的关键入口。

</SiteSection>
