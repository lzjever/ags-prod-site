# Audit

Audit 面向管理员回答几类最关键的问题：最近发生了什么变化，谁改了什么，哪些资源出现了异常，以及接下来应该去哪里处理。它不是为了堆更多日志，而是为了把项目里真正值得追责、复盘和继续处理的事件集中起来。

<SiteSection title="核心作用" description="Audit 用于把项目中的关键变更、异常与治理动作集中展示给管理员。">

Audit 会把项目中的关键变更、异常事件和治理动作集中展示给管理员，并通过主表、详情抽屉和筛选条件提供排查入口。这里出现的事件并不局限于“某个人改了配置”这类静态操作，还会覆盖 Endpoint 调用、资源策略变化、成员状态变化、智能体相关动作，以及 Notebook/Chat 相关的关键执行事件。它让管理者不需要依赖外部日志和零散记录，就能在产品内部建立一条稳定的回看路径。

</SiteSection>

<SiteSection title="为什么重要" description="没有 Audit，管理员就很难完整回看变化过程并进行排查。">

没有 Audit，很多变化就只能停留在“似乎发生过”，而很难被系统地回看和解释。有了这层能力，管理员更容易把 Usage、Policy 和执行结果放进同一个因果链里理解，也更容易完成项目级排查与审查。尤其在资源限制命中、成员权限变化或某次执行异常之后，Audit 往往是最先打开的页面，因为它最适合回答“到底发生了什么”。

</SiteSection>

<SiteSection title="典型使用场景" description="Audit 最适合在资源限制调整、异常排查和责任归属确认时使用。">

在典型场景里，管理员会通过 Audit 查看某次策略变更是谁发起、作用于哪个资源，再根据异常事件继续跳转到成员、策略或端点页面，并结合 Usage 与任务结果判断是否需要进一步治理动作。比如某个 Endpoint 调用突然报错、某个 Agent key 被创建、某次策略更新刚刚落下去，管理员都可以先从 Audit 找到事件，再决定后面去哪处理。

</SiteSection>

<SiteSection title="它和 Usage 的区别是什么" description="一个偏向事件回看，一个偏向资源消耗理解。">

Usage 更适合回答“资源用了多少、变化趋势如何”这类问题，Audit 更适合回答“到底发生了什么、谁触发了变化、异常出现在什么位置”这类问题。两者结合在一起，团队才既能看到消耗结果，也能理解导致这些结果的关键事件。

</SiteSection>
