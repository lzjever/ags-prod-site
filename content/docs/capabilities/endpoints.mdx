# Endpoints

Endpoints 是项目里所有模型资源的统一入口。无论成员在 Chat 中提问、在 Notebook 中运行任务，还是让智能体代为执行，系统都需要先知道“这次该调用哪一个模型、通过什么协议调用、凭据从哪里来、这个资源当前是否可用”。Endpoints 负责的，就是把这些问题固定成可以被反复引用和管理的项目对象。

<SiteSection
  title="它解决什么问题"
  description="如果模型配置散落在不同页面里，后续使用、限制和排查就很难对得上。"
>

很多团队在早期试用 AI 时，会把模型地址、供应商参数和密钥引用直接写在不同页面或脚本里。这样虽然能快速跑起来，但一旦项目进入多人协作，马上就会出现一连串问题：同一个模型到底被哪些页面在用、出了错误后应该查哪里、限额和费用该围绕哪个对象设置。Endpoints 的意义，就是把这些原本分散的模型配置收拢成统一入口，让后续所有使用和管理都围绕同一个对象展开。

</SiteSection>

<SiteSection title="一个 Endpoint 里通常会包含什么" description="它不是一句模型名，而是一份完整的接入定义。">

一个 Endpoint 通常会包含这些信息：它连接的是哪一家模型服务、使用什么协议、默认模型是哪一个、绑定的是哪一份凭据、当前状态是否正常，以及基础限额怎样设定。这样做的好处是，管理员不用每次都重新解释“这个模型入口到底是什么”，成员也不用猜“当前对话或任务背后实际调用的是哪一个模型资源”。

</SiteSection>

<SiteSection title="为什么它会影响后面的每一页" description="资源只有先成为正式对象，后续使用、限制和排查才真正能对应起来。">

当 Chat、Notebook 和 Agents 都围绕同一组 Endpoint 工作时，系统才能把这些问题接起来：这次请求到底用了什么资源，这个资源是否应该被限制，最近的消耗主要来自哪里，某次异常究竟发生在什么对象上。也正因为如此，Endpoints 页面对项目的意义并不只是“配置模型”，而是给后面的 Resource Policy、Usage 和 Audit 提供一个共同的参考点。

</SiteSection>
