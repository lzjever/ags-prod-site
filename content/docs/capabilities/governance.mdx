# Governance 总览

在 AgentSmith 中，治理不是独立于实际使用的后台能力，而是和对话、任务执行、智能体调用一起持续生效的主线。团队成员在项目中使用资源时，系统会同步判断资源是否可访问、限制是否命中、消耗如何累积，以及关键变化是否需要被记录下来。

<SiteSection
  title="治理围绕项目资源展开"
  description="它关注的是谁能使用什么资源、可以使用到什么程度，以及后续如何回看。"
>

AgentSmith 当前的治理主线围绕项目级 Endpoint 展开。Endpoint 定义模型资源，Resource Policy 决定访问与限制，Usage 用来观察实际消耗，Audit 用来回看关键事件与异常。因为这些能力都围绕同一对象工作，所以治理不会散成若干互不相关的设置页。

</SiteSection>

<SiteSection
  title="治理会在真实使用中生效"
  description="资源限制不是写在纸面上的规则，而是会进入调用和任务执行链路。"
>

当成员发起对话或运行 Notebook 任务时，系统会在真正开始执行之前检查资源是否可用、访问是否被允许、是否命中速率限制，以及是否触及支出限制。如果条件不满足，任务会被明确阻断，后续也会留下可回看的使用和审计记录。这意味着治理不是事后解释，而是事前和事中都真实参与使用过程。

</SiteSection>

<SiteSection
  title="治理的目标不是增加摩擦，而是让长期运行更可控"
  description="对企业团队来说，持续可见比单次放开更重要。"
>

企业真正需要的不是一个完全没有边界的使用环境，而是一套可以放心长期运行的系统。治理能力的意义，不在于把系统变复杂，而在于让团队知道资源如何被使用、限制何时生效、异常发生在哪里，以及后续应该到哪里回看。这些信息一旦连续可见，AI 才更容易进入稳定业务流程。

</SiteSection>

<SiteSection
  title="这条治理主线会和 Use 面持续对应"
  description="它不会脱离项目上下文单独存在。"
>

Chat、Notebook、Agents 和 Files 解决的是“如何把工作做起来”，Governance 解决的是“如何让这套工作长期可理解、可限制、可追溯”。两者围绕同一个项目继续发生，因此使用体验和治理边界不会彼此割裂。

</SiteSection>
