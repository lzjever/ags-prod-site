# 核心能力总览

在 AgentSmith 里，这些能力不是很多模块的平铺，而是同一套工作过程里的不同环节。团队通常先从 Chat 或 Notebook 开始，随后引用文件资料、调用智能体，再在同一个项目中查看资源限制、消耗变化和审计记录。对用户来说，这些页面虽然分布在不同栏目里，但在日常工作中往往会连续出现。

<ValueStatement
  eyebrow="Capability Structure"
  title="每一项能力，都会把工作继续带向下一步。"
  summary="Chat 负责快速开始，Notebook 负责推进复杂任务，Files 负责留下资料和结果，Agents 负责接入执行能力，Governance 负责回答“谁能调用、调用了多少、出了问题去哪查”。"
  detail="它们不是彼此分离的功能，而是同一套系统里的不同分工。"
/>

<SiteSection
  title="先开始工作，再把工作继续推进"
  description="Use 面不是一组孤立工具，而是一条从轻量交互到复杂任务的连续路径。"
>

Chat 让团队在项目里快速开始工作，适合提问、讨论和连续对话；Notebook 则让更复杂的任务拥有明确的输入、过程和结果，适合长期推进和协作接手；Files 负责把资料和结果留下来，避免输入和产出散落在临时目录或个人环境中。对很多团队来说，这三者共同决定了 AI 使用能否从“偶尔可用”变成“真的能融入日常工作”。

</SiteSection>

<SiteSection
  title="把执行能力从个人能力变成团队能力"
  description="智能体只有被正式接入项目，团队成员才能知道它在哪里、什么时候该用它。"
>

Agents 模块负责把外部智能体或平台托管智能体接入系统，使执行能力不再依赖某个人的本地环境。任务在选择智能体后，系统会继续准备执行、接收过程回传并保留结果。这样一来，智能体能力就不再只是“谁能在自己电脑上跑起来”，而会变成“团队知道在哪里调用、调用后结果留到哪里”的正式能力。

</SiteSection>

<SiteSection
  title="治理不是另一个后台，而是实际使用的一部分"
  description="资源配置、访问限制、用量记录和审计回看会跟着真实调用一起发生。"
>

当成员在 Chat 或 Notebook 中调用资源时，系统不会把治理留在另一套后台里。Endpoints 负责定义模型资源，Resource Policy 决定访问和限制，Usage 反映真实消耗，Audit 记录关键变更与异常。这样一来，团队看到的不只是“有没有限制”，而是每一次调用如何发生、消耗如何变化、异常发生后能否被快速说明。

</SiteSection>
