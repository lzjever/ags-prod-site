# Resource Policy

Resource Policy 负责围绕 endpoint 资源定义访问控制与 rate / spending 限制。

<SiteSection
  title="治理模型"
  description="当前策略围绕 project default、resource override 与 subject override 组织。"
>

当前策略模型围绕 `Project Defaults`、`Resource Override` 和 `Subject Override` 组织，并最终共同解析为 effective policy。这样的设计让默认规则、资源级特例和面向主体的差异化约束可以同时存在，也让治理结果更容易被系统化解释。

</SiteSection>

<SiteSection
  title="使用影响"
  description="管理员不仅要看到最终结果，也要能理解当前策略如何作用于真实资源使用。"
>

管理员不仅要看到最终结果，还要理解当前策略如何作用于真实资源使用。多层覆盖的优先级需要被解释清楚，策略配置需要在真实使用路径中被执行，治理结果也需要在 Audit 和 Usage 里被验证。

</SiteSection>

<SiteSection title="它如何影响实际使用" description="Resource Policy 不是纸面规则，而是直接影响成员在项目中的资源使用方式。">

Resource Policy 会直接决定成员能否访问某个端点、请求频率和费用边界如何设置，也会影响 Audit 和 Usage 中看到的治理结果是否足够可解释。换句话说，它不是离业务很远的一层，而是项目资源使用方式最直接的塑形者。

</SiteSection>

<SiteSection title="为什么它不是单纯的限制页" description="真正的价值在于让限制与后续事实保持一致。">

如果一套策略只能被配置，却无法在真实调用中稳定生效，也无法在后续记录里被回看，它就很难真正帮助企业管理资源。Resource Policy 的价值，在于它不仅决定规则，还会影响任务是否被允许启动、请求是否被限流，以及后续 Usage 和 Audit 里会出现什么样的事实记录。这样一来，限制就不再只是纸面规则，而是进入真实使用主线的一部分。

</SiteSection>
