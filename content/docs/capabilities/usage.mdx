# Usage

Usage 的目标很克制：回答用户“我在用哪些资源、用了多少、离上限还有多远”。它不是给管理员做深度排障的日志页，而是把和资源消耗相关的事实用最容易理解的方式展示出来，让成员和管理员都能尽快判断当前状态是不是正常。

<SiteSection title="产品边界" description="Usage 面向普通成员，不承载管理员排障与复杂诊断。">

Usage 面向的是普通成员，但管理员也会频繁使用它。它需要克制地展示资源消耗、额度进度和趋势变化，让用户能快速理解当前状态，而不会被复杂诊断信息打断。对一个正在稳定使用 AI 的团队来说，这一页最重要的意义不是“图表做得多复杂”，而是让人一眼就看清：哪个 Endpoint 最近最忙、哪个额度快到上限、最近的变化是不是异常。

</SiteSection>

<SiteSection title="为什么它对日常使用很重要" description="它帮助用户在被拒绝前就调整行为，而不是等错误发生后才知道超限。">

它的重要性在于，让用户在真正被拒绝之前就能感知变化并调整行为。无论是空数据、接近上限还是已经命中限制，Usage 都应该用低心智的方式把状态表达清楚，而不是把管理员视角的诊断信息强塞给普通用户。当前系统也已经围绕项目用量做了真实持久化和聚合，因此这里看到的不只是临时页面状态，而是围绕 Endpoint 汇总出来的实际消耗事实。

</SiteSection>

<SiteSection title="它和 Audit 的区别是什么" description="一个回答“用了多少”，另一个回答“发生了什么”。">

Usage 更适合回答“这几天哪个 Endpoint 消耗上升最快”“今天和昨天相比变化大不大”“某个额度是不是快到上限”这类问题。Audit 更适合回答“是谁改了资源策略”“哪次调用触发了异常”“某个限制是什么时候开始生效的”。两者一起看时，团队才既能知道结果，也能知道结果背后的原因。

</SiteSection>
