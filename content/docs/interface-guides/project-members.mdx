# 成员治理

<SiteSection title="你会在这里做什么" description="这里用于管理成员、加入申请与访问状态。">

成员治理页把“谁可以进入项目、进入之后能做什么”这件事讲得非常具体。管理员可以在这里查看成员状态、处理加入申请、调整分组，并核对某个人当前的有效访问范围。这样的设计让权限管理不再停留在抽象角色定义上，而是直接落到每一个真实成员身上。

对团队来说，这种清晰感很重要，因为协作的边界一旦模糊，模型、文件和任务就很难真正做到可控。

</SiteSection>

<SiteSection title="关键看点" description="成员治理页最适合用来讲清楚项目级权限边界如何落到具体成员身上。">

这张页面的价值在于，它把治理从“后台配置”推进到了“日常协作”。成员状态、加入申请和分组都集中呈现，意味着权限不再只是系统内部的一套规则，而是真正影响每个人在项目里能看到什么、能调用什么、能参与哪些任务的业务边界。

</SiteSection>

<SiteSection title="关联模块" description="成员治理与资源策略、审计和实际使用行为天然相连。">

成员分组会进一步影响资源策略，成员变更又会进入 Audit 留下回看依据，而最终生效的访问范围还会直接体现在 Chat、Notebook、Files 等页面的实际使用体验中。因此，成员治理从来都不是一张孤立的后台页面，它始终和业务使用面紧密相连。

</SiteSection>
