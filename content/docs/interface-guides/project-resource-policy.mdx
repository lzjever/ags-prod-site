<SiteSection title="这个页面的作用" description="资源策略页负责把资源限制变成明确且可执行的项目规则。">

团队一旦开始稳定使用模型资源，就不可能只依赖默认放开。谁可以访问哪些资源、调用频率需要控制到什么程度、费用边界如何定义，这些问题都必须被明确下来。资源策略页承担的，就是把这些限制写成项目内真正会生效的规则，而且这些规则不只作用于 Endpoint，也会覆盖当前支持范围内的文件库和智能体资源。

</SiteSection>

<SiteSection title="你通常会在这里做什么" description="查看规则、调整限制和确认不同主体的可用范围，是这个页面最常见的动作。">

管理员通常会在这里查看项目默认规则、某个资源的特殊规则，以及某个成员或成员分组的例外限制，并根据实际资源使用情况继续调整访问范围、限流或费用边界。页面里真正重要的，不是把规则写得越多越好，而是让人一眼就看懂：默认开放到什么程度，哪个资源被单独收紧了，某个成员为什么比其他人多一层额外限制。

</SiteSection>

<SiteSection title="为什么它重要" description="真正有价值的限制，必须能在调用时拦得住，也能在后面查得到。">

资源策略页的重要性，不在于列出了多少规则，而在于这些规则会继续进入任务和调用链路。当成员发起对话或任务时，系统会基于这里的设置判断是否允许执行；当限制生效或被命中时，后续又能在 Usage 和 Audit 中继续看到相关记录。这样一来，管理员看到的就不只是“设置过规则”，而是“哪条规则拦下了哪次调用，这次限流或超限是不是符合预期，后面该去 Usage 还是 Audit 继续排查”。

</SiteSection>
