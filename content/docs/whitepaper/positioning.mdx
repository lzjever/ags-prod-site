# 定位与范围边界

AgentSmith 的定位可以概括为一句话：它是一套帮助企业把 AI 使用、智能体执行和资源治理放进同一项目工作流的产品。它不是只负责一段对话，也不是只负责资源设置，而是试图把“使用、执行、结果、治理、追溯”组织成一条连续主线。

这意味着它既不是普通意义上的 AI 聊天产品，也不是只给管理员使用的治理后台。对于企业团队来说，真正需要的不是若干孤立功能，而是一套既能让一线成员用起来、又能让管理者长期掌握边界和事实的产品结构。

<ValueStatement
  eyebrow="Positioning"
  title="这套产品的成立前提，是把“能用起来”和“能长期运行”同时放进设计中心。"
  summary="很多产品擅长解决一个切面的问题，但企业真正需要的是一套能同时承接使用体验、治理边界、执行结果和后续追溯的结构。"
  detail="AgentSmith 的定位之所以成立，正是因为它没有把这些问题拆成几套互相松散的工具。"
/>

<SiteSection
  title="适合什么样的场景"
  description="它更适合已经准备让 AI 进入持续工作流的团队，而不只是一次性试用。"
>

如果一个团队已经开始面向真实业务使用 AI，希望把对话、任务、文件和执行能力放进统一项目环境，希望不同成员围绕同一份上下文协作，也希望资源限制、用量记录和审计回看真实进入日常工作，那么 AgentSmith 会比单点工具更合适。它尤其适合那些不仅关心“这次能不能跑出来”，也关心“结果能否留下、过程能否解释、资源能否被管理”的组织。

</SiteSection>

<SiteSection
  title="当前明确在范围内的内容"
  description="以下能力共同构成了当前产品主线。"
>

当前范围内的内容包括系统管理、工作区生命周期、项目级使用面与治理面。具体来说，Chat、Notebook、Files、Agents 构成日常使用与执行主线，Endpoints、Resource Policy、Credentials、Members、Usage、Audit 和 Settings 构成治理主线。围绕 Notebook 执行产生的任务工作目录、输入清单、凭据文件、trace 与 artifacts，也属于当前主线的一部分，因为它们决定了 AgentSmith 是否能成为真正可落地的运行环境。

</SiteSection>

<SiteSection
  title="当前不打算解决什么"
  description="边界清晰，本身就是产品成熟度的一部分。"
>

AgentSmith 当前并不打算成为通用 DevOps 发布编排平台，也不打算把产品主线扩展为脱离项目上下文的组织级总控系统。它也不会把所有治理问题都扩张到文件级、组织级或跨项目无限上卷。当前重点始终是围绕 `Workspace -> Project -> Use / Govern -> Execution / Evidence` 这条主线把产品做实，而不是把边界不断拉宽。

</SiteSection>

<SiteSection
  title="用户应当如何理解它的边界"
  description="理解边界，有助于更准确地判断这套产品是否适合自己的团队。"
>

如果你需要的是一个轻量的个人 AI 助手，那么 AgentSmith 可能不是最合适的形态；如果你需要的是一套能让团队协作、资源治理、执行结果和后续回看都围绕同一项目继续发生的系统，那么它的价值就会更明显。换句话说，它服务的是企业团队的长期工作，而不是一次性的个人体验。

</SiteSection>
