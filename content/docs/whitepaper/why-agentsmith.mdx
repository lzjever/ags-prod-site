# 为什么是 AgentSmith

AgentSmith 的回答不是再做一个新的 AI 入口，而是把企业最容易断开的几段链路重新接起来。它把业务空间、项目容器、日常使用、智能体执行、文件沉淀、资源治理和证据回看组织进同一套系统里，让团队不必再靠人工拼接一整条工作链。

<ValueStatement
  eyebrow="Structural Answer"
  title="AgentSmith 的差异，不在某一个页面，而在于它把原本分散的问题收成了一套连续结构。"
  summary="团队进入系统、创建项目、发起对话、推进任务、沉淀结果、查看用量和回看审计，都围绕同一组对象继续发生。"
  detail="这样一来，AI 使用就不再只是一次次独立调用，而会逐步变成一条稳定的业务主线。"
/>

<SiteSection
  title="它先给团队一个稳定容器"
  description="Workspace 和 Project 不是导航层级，而是稳定边界。"
>

AgentSmith 先用 Workspace 建立业务空间，再用 Project 承接真正的工作对象。对话、任务、文件、智能体、模型资源、成员权限、用量和审计记录都围绕同一个 Project 持续保留。这使得团队不必在不同后台之间反复切换，也不必在每次使用时重新建立上下文。

</SiteSection>

<SiteSection
  title="它把一次次使用组织成连续任务"
  description="对话适合快速开始，Notebook 适合把复杂工作持续推进。"
>

Chat 负责降低起步门槛，让成员能够快速提问和持续沟通；Notebook 则负责把复杂执行升级成长期任务。输入资料、消息过程、trace、产物文件和最终状态都会保留在同一任务中，因此任务可以被继续推进、被其他成员接手，也可以成为后续工作的输入。这使得复杂工作不再停留在临时会话和个人终端里。

</SiteSection>

<SiteSection
  title="它让资料和结果真正留下来"
  description="Files 不是附件区，而是项目资产层。"
>

AgentSmith 让项目资料、任务输入和执行结果在同一套文件资产层中持续组织。文件可以成为任务上下文，任务结果也可以继续回流到项目文件库中。这样一来，AI 结果不会再只是一段被看过的回复，而会逐步变成团队可以继续使用的项目资产。

</SiteSection>

<SiteSection
  title="它把执行能力纳入正式工作流"
  description="智能体不再只是个人环境里的隐性依赖。"
>

Agents 模块把外部智能体和平台托管智能体都纳入项目正式对象。团队成员可以在任务中直接选择执行能力，系统会在运行前检查资源、权限和限制，在运行中持续回传状态、trace 和产物，在运行后再把结果继续留在项目历史中。智能体因此不再只是某个人维护的外部脚本，而会逐步变成团队可以共同使用和管理的能力。

</SiteSection>

<SiteSection
  title="它让治理真正进入使用主线"
  description="资源定义、限制、生效结果和后续回看都围绕同一条链路继续发生。"
>

在 AgentSmith 中，治理不是与日常使用分离的后台附属页。Endpoint 定义模型资源，Resource Policy 控制访问和限制，Usage 记录实际消耗，Audit 记录关键事件和异常。更重要的是，这些规则会在真实任务和真实调用中生效，而不是只停留在配置页面里。企业因此不仅能配置资源，也能真正理解资源是如何被使用和被限制的。

</SiteSection>
