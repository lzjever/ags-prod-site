# AgentSmith 产品网站开发说明书

更新日期：2026-03-17  
状态：`draft-v0.1`

## 1. 文档目的

本说明书用于约束 `ags-prod-site` 的开发方式、内容组织方式和后续建站实施路径。

它服务三个目标：

1. 让本项目从一开始就有清晰的素材同步和内容治理规则。
2. 让后续站点实现不偏离 AgentSmith 的真实产品边界。
3. 让网站开发、内容编排和素材维护能够持续进行，而不是一次性拼装。

## 2. 当前项目状态

当前仓库是空目录起步，后续会承接：

1. Next.js 站点工程
2. 参考资料副本
3. 产品白皮书与功能说明内容
4. 截图、图片和站点资源

因此当前阶段必须先完成基础规则和目录约定，再开始建站实现。

## 2.1 当前推荐技术方案

本项目当前推荐方案固定为：

`Next.js App Router + MDX + Fumadocs + 自定义视觉组件 / 自定义营销页面`

选择这套方案的原因是：

1. `Next.js` 适合承接高自由度首页、专题页、动画页和组件化视觉表达。
2. `MDX` 适合承接白皮书、功能说明、架构说明和截图说明等内容型页面。
3. `Fumadocs` 适合作为文档内容结构层，而不是把整站锁死在标准 docs theme 里。
4. 这套组合更适合“产品白皮书 + 功能说明 + 市场推广 + 强视觉表达”的混合型站点。

当前不作为首选方案：

1. `Nextra`
2. `Docusaurus`
3. `Astro Starlight`

原因不是它们不能做，而是当前项目对首页叙事、动效表达、截图驱动视觉和自定义营销页面的要求更高。

## 3. 必做前置动作

在开始任何内容编排或页面实现前，先同步参考资料：

```bash
./scripts/sync-reference-sources.sh
```

可选复制全部截图归档：

```bash
./scripts/sync-reference-sources.sh all
```

同步后，本地参考目录约定为：

1. `references/prd-v1/`
2. `references/product-docs/latest/`
3. `references/product-docs/all/`（仅 `all` 模式）

## 4. 内容来源治理规则

### 4.1 一等来源

以下内容可以直接作为站点事实来源：

1. `references/prd-v1/`
2. `references/product-docs/latest/`
3. `../agentsmith` 中可验证的 README、开发文档、contracts、tests、页面实现

### 4.2 二次加工原则

站点文案允许重写，但必须满足：

1. 语义更清楚，不改变事实。
2. 产品表达更友好，不弱化边界。
3. 不把“规划中”写成“已支持”。
4. 不把工程内部术语直接当默认产品语言。

## 5. 建站目标结构

推荐把站点组织为如下层次：

1. `首页`
2. `产品白皮书`
3. `产品架构`
4. `核心能力`
5. `通用智能体运行链路`
6. `界面与功能说明`
7. `版本与参考资料`

推荐的内容关系是：

1. 首页负责一句话定义、价值摘要和核心导航。
2. 白皮书负责高层叙事。
3. 产品架构负责对象模型、层级结构和任务链路。
4. 核心能力负责按模块解释功能。
5. 页面说明负责接住截图和页面证据。

## 6. 站点实施原则

### 6.1 工程目标

后续站点工程应满足：

1. 文档驱动内容管理。
2. 结构清晰的侧边导航。
3. 便于插入截图、流程图和高层说明。
4. 支持后续快速新增章节和页面。

### 6.1.1 技术落地方式

推荐把工程分成两个协作层：

1. `内容层`
   使用 `MDX + Fumadocs` 管理白皮书、说明文档、模块介绍和截图说明。
2. `视觉层`
   使用 `Next.js App Router + 自定义 React 组件` 管理首页、专题页、Hero、动画模块和营销化编排页面。

换句话说：

1. `Fumadocs` 负责“文档结构和阅读体验”。
2. `Next.js 自定义页面` 负责“品牌表达、视觉冲击和推广转化”。
3. 两者共享同一套站点样式、组件和内容资产。

### 6.2 内容表现原则

1. 首页强调产品定位，不做模板化博客首页。
2. 页面说明优先“问题 - 功能 - 用户操作 - 证据”结构。
3. 截图与说明需要配对呈现。
4. 白皮书章节与功能说明章节要分层，不混写。
5. 首页、专题页和功能亮点页允许明显区别于文档页的视觉编排方式。

### 6.2.1 最终站点语系原则

最终面向访客的网站文案，必须使用 `产品理解语系`，不能混入 `建站过程语系`、`内容治理语系` 或 `内部协作语系`。

更严格地说：任何公开页面都不应出现“我们如何理解这个产品、如何组织这些内容、这页想说明什么、这项能力值得单独讲、适合谁拿去讲解”这类 `产品策划语系` 或 `内容导览语系`。

必须坚持以下规则：

1. 句子的主语优先是 `产品`、`能力`、`用户动作`、`业务场景` 或 `结果价值`，而不是 `站点`、`页面组织方式`、`文档结构`、`截图处理方式`。
2. 最终页面文案应回答“这个产品做什么、为什么重要、怎么使用、带来什么结果”，而不是解释“我们如何搭建这个网站、如何组织这些内容、如何处理这些截图”。
3. 允许在内部治理文档中出现 `快照`、`同步`、`引用源`、`状态治理`、`站点结构` 等词，但这些词不应默认进入对外公开页面。
4. 允许在合适场景中提到 `真实界面`、`产品白皮书`、`架构说明` 等内容类型，但不能把它们写成“站点制作说明”式话术。
5. 图片上的标签、说明卡片、局部注释、动效配文，也必须遵守同样的产品语系规则，不能写成“这里做了裁切 / 聚焦 / 放大 / 重构”这类制作过程说明。
6. 页面不应自我解释。例如，不应写“这页最想讲清什么”“为什么这项能力值得单独讲”“如果首页回答的是……这里回答的是……”这类解释内容编排方式的句子。
7. 页面不应以我们团队的使用场景为中心。例如，不应写“适合售前讲解”“便于内部对齐”“帮助产品评审快速理解”这类面向内部使用方式的句子，除非该页面本身就是内部文档，而不是公开站点内容。

明确禁止进入最终站点的表达倾向：

1. `首页主视觉现在直接使用真实界面裁切后的关键区域` 这类解释制作方式的句子。
2. `关键卖点都能落到真实界面与说明页，不把白皮书和推广表达做成两套互相脱节的话术` 这类解释内容治理方式的句子。
3. `用于对外讲解 / 内部对齐 / 证据支撑 / 第一版内容入口 / 站点围绕...组织` 这类偏内部协作和内容编排视角的句子。
4. `证据页`、`界面证据`、`文档镜像`、`结构化入口` 等更适合内部策划讨论、但不适合作为公开页面主叙事的表述。
5. `为什么这项能力值得单独讲`、`这页最想传达的判断`、`如果你更关心……建议先看……`、`给售前与交付` 这类内容导览或产品策划话术。

页面上线前必须做一次语系审查：

1. 如果读者完全不知道我们的建站过程，这句话是否仍然自然成立。
2. 如果删掉“站点 / 页面 / 文档 / 截图 / 说明页 / 白皮书”这些词，句子是否仍能表达清楚产品价值。
3. 这句话是否在讲产品本身，而不是在讲我们如何组织对产品的表达。
4. 这句话是否更像产品官网文案，而不是项目复盘、PR 说明或开发注释。
5. 这句话是否在向潜在客户和用户介绍产品，而不是在向我们自己解释产品结构。

当前经验结论：

1. 当团队同时维护白皮书、文档、截图说明和营销页面时，最容易出现“内部治理语言泄漏到最终站点”的问题。
2. 视觉增强和截图重构之后，尤其容易顺手写出“裁切、聚焦、说明页、证据页”这类制作视角文案，必须刻意避免。
3. 任何首页、专题页、截图注释和导览文案，都应优先回到用户价值、产品能力和使用场景本身。
4. 任何“我们如何理解产品”“这页想说明什么”“为什么值得这样组织”的句子，都应优先改写成直接面向访客的产品说明。

### 6.3 视觉方向

产品网站应该体现“企业级控制平面 + 智能体运行环境”的气质：

1. 结构理性、信息密度适中。
2. 视觉上偏产品叙事与技术可信感，不走夸张营销风。
3. 截图、流程图、信息卡片要共同服务理解，不只是装饰。

### 6.3.1 与 AgentSmith 主产品的视觉协调原则

产品网站的 UI 元素、配色和质感应尽可能与 `agentsmith` 主产品保持一致协调。

这不意味着机械复刻后台界面，而是要求站点在视觉语言上与主产品同源，包括：

1. 背景层级应延续主产品的深色体系与分层逻辑。
2. 强调色优先沿用主产品的品牌蓝与 AI 渐变方向。
3. 卡片、边框、按钮、对话框和信息模块的质感应与主产品协调。
4. 字体、圆角、间距密度和交互反馈应尽量延续主产品的专业工具感。

当前可直接参考的主产品视觉基因包括：

1. 基础背景：`#121214` / `#17181B` / `#1F2125`
2. 高亮面：`#2B2E34` / `#383C44`
3. 品牌强调色：`#7CA0FF`
4. AI 渐变方向：浅蓝到高饱和蓝
5. 文字层级：强文本白色，正文冷灰白，次级文本蓝灰

站点实现时的推荐策略：

1. 以主产品色板为基础建立站点 token。
2. 在首页和专题页中可适度提升层次感和戏剧性，但不脱离主产品气质。
3. 不引入与主产品冲突的主色系，不改成浅色品牌站或泛紫色 AI 模板风。

### 6.4 视觉目标升级

本项目的视觉目标不是“合格的文档站”，而是“顶尖的产品白皮书与产品推广网站”。

这意味着站点必须同时满足：

1. 有足够强的第一眼品牌与产品气质。
2. 有足够清晰的结构化信息承载能力。
3. 能把复杂产品讲清楚，同时把核心亮点讲得有冲击力。
4. 既适合深入阅读，也适合演示、售前讲解和市场传播。

### 6.5 截图驱动的视觉表达原则

我们已经拥有较完整的系统截图、截图说明和索引信息，因此截图不是补充材料，而是核心视觉资产。

后续站点设计应优先考虑：

1. 从现有系统截图中提炼高质量 hero 视觉和模块展示视觉。
2. 对截图进行裁切、分层、放大、局部高亮和组合排版，以突出关键功能。
3. 将截图与功能标签、价值文案、流程箭头、状态标识结合，形成“解释型视觉”。
4. 避免把截图简单堆成图库；每张关键截图都应服务一个清晰卖点。

补充要求：

1. 在正式使用截图前，应先对原图进行人工观看和内容理解，而不是机械整图缩放后直接上页面。
2. 如果整图缩放后信息过密、关键区域不清晰或视觉焦点不明确，应优先进行二次编排，而不是保留低质量呈现。
3. 允许基于真实截图做自由裁切、局部放大、聚焦重构、分层拼贴和视觉节奏重排。
4. 首页、专题页和亮点模块中的截图默认不要求保持“整图原样平铺”，而要求优先服务阅读清晰度和视觉表达质量。
5. 只要不改变产品事实，就可以对截图做合理的视觉强化，包括聚焦、透视编排、局部动画和滚动联动展示。

### 6.6 动画与动态效果原则

允许并鼓励基于已有截图和页面结构设计高质量动态效果，用于强调核心功能和功能亮点。

可采用的动态表现包括：

1. 截图分层浮动、视差和渐进揭示。
2. 基于页面区域的局部放大、高亮扫描和焦点切换。
3. 核心任务流的滚动联动动画。
4. Chat、Notebook、Files、Govern 之间关系的结构化转场。
5. 关键数据卡片、流程卡片和状态标签的节奏化入场动画。
6. 基于截图局部的裁切切换、焦点移动和多视图拼接动画。

动画必须遵守以下约束：

1. 优先服务理解，而不是炫技。
2. 节奏克制，避免廉价感和过度干扰阅读。
3. 桌面端与移动端都必须保持可用。
4. 必须兼顾性能、降级方案和可访问性。

### 6.7 视觉设计责任

本项目后续页面设计必须主动承担以下职责：

1. 强化产品主叙事，而不是只做页面容器。
2. 把 `企业级 AI 控制平面` 与 `通用智能体运行环境` 两条主线做出差异化视觉表达。
3. 让读者在视觉上直接感知 `System / Workspace / Project / Use / Govern` 的层级关系。
4. 让 Chat、Notebook、Files、Agents、Endpoints、Usage、Audit 等能力有清晰且高级的模块识别度。

### 6.8 禁止的视觉退化

后续建站过程中，明确禁止以下做法：

1. 直接套用默认文档框架主题后只改文字。
2. 把首页做成普通文档索引页或博客列表页。
3. 把所有截图等比例平铺，缺少主次关系和讲述结构。
4. 用过多通用 UI 组件堆砌出“AI 模板站”风格。
5. 只追求动效数量，不服务卖点表达和阅读效率。

## 7. 推荐目录规划

后续工程建议至少包含：

1. `docs/`：本项目治理文档
2. `references/`：同步后的参考资料副本
3. `scripts/`：资料同步和构建辅助脚本
4. `app/`：Next.js 路由页面与专题页
5. `content/`：MDX 文档内容与站点内容源
6. `components/`：文档站与营销页共享组件
7. `public/`：截图、静态资源、图示、动画素材
8. `lib/`：内容装载、文档索引和展示辅助工具

在真正初始化站点工程前，不必过早固定全部技术目录，但要先固定参考资料和治理文档目录。

## 8. 内容建模建议

后续页面内容建议按以下模型组织：

### 8.1 概述型页面

适用于首页、产品摘要、白皮书总述。

推荐结构：

1. 一句话定义
2. 解决的问题
3. 核心价值
4. 结构总览
5. 继续阅读入口

### 8.2 模块型页面

适用于 Chat、Notebook、Files、Agents、Endpoints、Audit 等页面。

推荐结构：

1. 模块定位
2. 解决的问题
3. 用户如何使用
4. 与其他模块的关系
5. 当前实现状态
6. 页面截图或证据

### 8.3 证据型页面

适用于截图目录、页面说明、功能对照。

推荐结构：

1. 页面标题
2. 路由或场景
3. 功能说明
4. 页面内容说明
5. 用户操作
6. 截图

## 9. 与主项目的关系约束

`ags-prod-site` 是 `agentsmith` 的配套产品站，不是独立重新定义产品的子项目。

因此必须遵守：

1. 产品网站的术语边界服从 `agentsmith` 的 PRD 和宪法。
2. 产品网站的截图与页面说明服从最新有效的 `product-docs`。
3. 如代码、测试、PRD 发生冲突，应显式记录差异后再决定如何呈现。

## 10. 更新机制建议

每次网站内容迭代建议遵循：

1. 先同步参考资料。
2. 再确认最新产品基线和截图归档。
3. 再修改站点内容与结构。
4. 最后检查术语、截图链接和页面状态是否一致。

## 11. 第一阶段开发任务定义

当前第一阶段聚焦“建站前治理与规划”，包含：

1. 参考资料同步脚本
2. 项目宪章
3. 开发说明书
4. 站点内容地图与页面优先级

本阶段暂不要求：

1. 完成站点工程初始化
2. 完成全量页面编写
3. 完成视觉精修

当前配套文档：

1. `docs/产品网站-项目宪章.md`
2. `docs/产品网站-开发说明书.md`
3. `docs/产品网站-内容地图与页面优先级.md`
4. `docs/产品网站-首页策划与二级目录树.md`
5. `docs/产品网站-受众分层与阅读路径.md`
6. `docs/产品网站-成功标准与验收口径.md`
7. `docs/产品网站-截图与视觉资产使用规范.md`
8. `docs/产品网站-内容状态治理与更新规则.md`
9. `docs/产品网站-视觉策略与Design Token草案.md`

## 12. 后续建议实施顺序

建议下一步按以下顺序推进：

1. 基于本说明书确定网站一级信息架构
2. 初始化 `Next.js + MDX + Fumadocs` 工程骨架
3. 先完成首页和产品白皮书主干页面
4. 再完成核心模块页与专题页
5. 导入截图证据页、静态资源与视觉组件
