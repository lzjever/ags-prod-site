# 12. 非功能需求、风险与验收标准

## 12.1 为什么这一章重要

对 AgentSmith 这样的系统来说，很多关键质量并不体现在页面上，而体现在：

1. 权限是否可信
2. 多租户边界是否可信
3. 执行结果是否可追溯
4. 错误是否稳定可诊断
5. 规划能力是否建立在稳固基础上

因此，非功能需求不是补充条款，而是产品可信度的一部分。

## 12.2 非功能需求

### 12.2.1 安全与权限

1. 后端必须是权限与策略判断唯一权威。
2. 前端 permission gate 只负责 UX，不替代安全。
3. Secret 不得在日志、trace、artifact 中泄露。
4. Agent 执行链路中的 bearer token 仅允许内存态传递，不允许持久化。
5. 用户级外部凭据与项目级凭据应明确边界，避免混用。

### 12.2.2 多租户隔离

1. workspace 必须是明确租户边界。
2. 所有 workspace 私有数据最终都应执行 tenant 级命名隔离。
3. system admin 与业务入口必须分离。
4. public workspace 可见性必须与 provisioning status 一致。

### 12.2.3 可审计性

1. 关键治理动作必须可追踪 actor、时间、目标对象、前后差异。
2. policy、usage、audit、workspace provisioning 需要有证据记录。
3. Notebook 执行必须具备 trace 与 terminal fallback。
4. 关键错误应能关联到 request / task / run 上下文。

### 12.2.4 可用性与可恢复性

1. 关键失败必须 fail-fast，并返回稳定错误码。
2. external / internal agent 执行失败必须能映射到可理解的前端提示。
3. workspace provisioning 失败必须可见、可重试、可追溯。
4. 长链路执行中断时，不应让用户落入无状态、无反馈的空白状态。

### 12.2.5 可演进性

1. 新能力优先复用现有权限、资源和协议结构。
2. 不在没有合同和边界说明的前提下引入新治理对象。
3. 技术实现应为 sandbox、skill registry、catalog 运营等未来能力预留明确扩展点。

## 12.3 当前主要风险

| 风险 | 影响 | 当前状态 |
|---|---|---|
| workspace provisioning 仍未完全打通数据面 | workspace ready 语义可信度受限 | 高 |
| tenant isolation 仅部分覆盖 | 多租户安全叙事仍需补强 | 高 |
| internal sandbox 依赖外部环境 | internal agent 托管能力不稳定 | 中高 |
| secret handling 仍属 MVP 级 | 合规与安全风险 | 中高 |
| `project:manage` 历史兼容残留 | 权限模型可能继续漂移 | 中 |
| system info 被误扩展为大盘 | 产品边界漂移 | 中 |

## 12.4 风险应对思路

### 风险一：workspace provisioning 语义不够强

应对方向：

1. 强化 publish -> foundation bootstrap -> status feedback 闭环
2. 让 `ready` 真正代表“可访问且已初始化”

### 风险二：tenant isolation 不完整

应对方向：

1. 盘点所有 workspace 私有 store
2. 逐个完成 tenant prefix / namespace 接入
3. 用集成验证证明隔离生效

### 风险三：internal sandbox 不稳定

应对方向：

1. 强化 sandbox manager readiness、startup、keepalive、release
2. 明确失败原因与错误映射
3. 用真实环境 smoke 验证主链路

## 12.5 验收标准

### 12.5.1 功能验收

1. 关键页面与主链路具备真实后端与 mock lane 双重可验证性。
2. Chat / Notebook / Files / Endpoints / Policy / Usage / Audit 主链路可跑通。
3. workspace 从 draft 到 ready/failed 的流程状态可见。
4. 关键权限分层符合 owner/admin/creator/admin 的目标边界。

### 12.5.2 合同验收

1. OpenAPI / AsyncAPI 与代码一致。
2. frontend-backend gating matrix 与权限实现一致。
3. agent execution protocol 与前后端消费模型一致。

### 12.5.3 质量验收

1. typecheck 通过
2. contracts check 通过
3. 关键单元测试与集成测试通过
4. 核心 E2E smoke 通过

### 12.5.4 体验验收

1. 用户在关键失败场景下能得到可理解反馈。
2. 管理员能区分“配置存在”和“能力真正可用”。
3. 使用面不会因为治理能力存在而变得晦涩难用。

## 12.6 PRD 级完成定义

当以下条件满足时，可认为 AgentSmith 从“骨架成熟”进入“平台成熟的下一阶段”：

1. workspace provisioning 具备真实 backend foundation 闭环。
2. tenant isolation 覆盖主要 workspace 私有数据面。
3. internal agent sandbox 形成稳定可运维路径。
4. skill 机制从 builtin 扩展到可治理、可兼容的生态形态。
5. model catalog、policy、usage、audit 三条治理线形成更完整的运营闭环。

## 12.7 本章结论

AgentSmith 当前最值得肯定的，不是“功能很多”，而是它已经形成了一条清晰而克制的产品主线：

`System Admin -> Workspace -> Project -> Use/Govern -> Evidence`

接下来最重要的，不是继续横向铺模块，而是把这条主线的控制面、数据面和执行面真正闭环。
