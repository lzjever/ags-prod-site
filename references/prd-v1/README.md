# AgentSmith 重构 PRD 文档 v1

## 文档定位

本目录不是技术笔记，也不是零散需求清单，而是一套面向产品、设计、研发、测试、交付共同协作的正式 PRD 文档集。

它服务于 AgentSmith 的两条并行且互相支撑的产品主线：

1. 企业级 AI 治理与控制平面
2. 通用智能体的易用且安全的运行环境与统一用户接口

文档基于 `agentsmith` 仓库截至 2026-03-17 的以下事实来源整理而成：

1. 仓库代码与目录结构
2. 白名单文档与项目宪法
3. contracts / UXUI / user-guides 等现行规范
4. 关键后端实现，包括 workspace provisioning、agent execution、notebook orchestration、skills、tenant isolation 等

特别说明：

本次 PRD 重构不再把 `sandbox`、`Notebook task runtime`、`文件库挂载`、`builtin skills` 仅仅视为底层实现细节，而是把它们提升为 AgentSmith 产品价值的一部分。因为对通用智能体而言，真正决定可用性的往往不是“模型能不能调用”，而是：

1. 能不能在统一界面里操作
2. 能不能减少环境配置负担
3. 能不能在更安全的运行环境中执行
4. 能不能把运行过程中的文件、产物和上下文沉淀为项目资产

## 编写原则

为了避免 PRD 失真、空泛或“像会议纪要”，本套文档统一遵循以下标准：

1. 不把未实现能力伪装成现状能力。
2. 不用只有术语和模块名的 bullet list 代替需求表达。
3. 每个核心章节尽量回答五类问题：
   - 为什么做
   - 给谁用
   - 解决什么问题
   - 如何使用
   - 如何验收
4. 关键章节同时包含：
   - 背景与问题定义
   - 用户与角色
   - 典型场景
   - 用户故事
   - 功能需求
   - 约束与边界
   - 验收口径
5. 对能力状态统一使用三种标记：
   - `已实现`
   - `部分实现`
   - `规划中`

## 为什么要加入 User Story

本项目适合把 `User Story` 纳入 PRD，而且非常有必要。

原因是：

1. AgentSmith 不是单一页面产品，而是系统管理、工作区、项目、使用面、治理面交织的控制平面。
2. 如果只写模块和字段，很容易看不出“谁在什么场景下为什么要用它”。
3. User Story 能帮助团队把产品结构、权限边界和任务链路连接起来。

但需要强调：

1. User Story 不是 PRD 的全部。
2. 它应当与任务流、规则、限制条件、异常场景和验收标准一起出现。

## 收尾说明

截至本轮重构完成，这套文档已经不再是“项目梳理笔记”或“功能清单扩写版”，而是一套具备以下特征的正式 PRD 初版：

1. 有明确产品定位、范围边界与路线图。
2. 有系统管理、工作区、项目、使用面、治理面的完整章节结构。
3. 有用户故事、关键任务流、异常场景和验收标准。
4. 有对当前实现、部分实现和规划中能力的清晰区分。
5. 有对 AgentSmith 双主线的统一表达：
   - 企业级 AI 治理与控制平面
   - 通用智能体的易用且安全的运行环境与统一用户接口

如果后续继续演进，建议优先沿着“版本规划、模块详细验收、依赖项矩阵、原型稿配套”四个方向继续深化，而不是重新推翻当前结构。
